# Cross-Chain Signature Replay in `changeRecipientAddress`

* **Severity**: Medium
* **Source**: [Sherlock](https://github.com/sherlock-audit/2022-09-harpie-judging/blob/main/004-M/004-m.md)
* **Affected Contract**: [Vault.sol](https://github.com/Harpieio/contracts/blob/97083d7ce8ae9d85e29a139b1e981464ff92b89e/contracts/Vault.sol#L67)
* **Vulnerability Type**: Signature Replay / Missing Domain Separation / Cross-Chain Authorization Reuse

## Summary

The `changeRecipientAddress()` function relies on an off-chain signature generated by Harpie's trusted `serverSigner` to authorize recipient address changes.

However, the signed message does **not include the blockchain's chain ID**, meaning the same signature can be reused on another network if the Vault contract is deployed there with the same configuration.

Although the contract contains replay protection through `_isChangeRecipientMessageConsumed`, that protection only exists within the storage of a single chain. As a result, a signature intended for one network can be replayed on another network before expiration.

## A Better Explanation (With Simplified Example)

### Intended Behavior

Harpie's backend generates a signature that authorizes a user to change their recipient address.

The signed message contains:

```solidity
keccak256(
    abi.encodePacked(
        msg.sender,
        _newRecipientAddress,
        expiry,
        address(this)
    )
)
```

The inclusion of `address(this)` ensures that a signature intended for one Vault contract cannot be reused on an entirely different contract.

The intended flow is:

1. User requests a recipient address change.
2. Harpie's server signs the authorization.
3. User submits the signed message.
4. Vault verifies the signature came from `serverSigner`.
5. Recipient address is updated.
6. Signature is marked as consumed.

### What Actually Happens (Bug)

The signed message includes:

* User address
* New recipient address
* Expiry timestamp
* Contract address

But it does **not** include:

```solidity
block.chainid
```

Because chain ID is missing, the exact same signed message can be valid on multiple blockchains.

If the Vault contract exists on Ethereum and Polygon with the same address and signer configuration, a signature generated for Ethereum can also be used on Polygon.

The replay protection mapping does not help because each chain maintains completely independent storage.

### Why This Matters

* Signatures become valid outside their intended security domain.
* Authorization intended for one blockchain can be reused on another.
* Replay protection only works per-chain, not globally.
* The protocol loses the guarantee that a signed action is restricted to a specific network.

While this issue does not directly allow theft or signature forgery, it violates an important security assumption surrounding authorization scope.

### Concrete Walkthrough (Alice & Harpie)

#### Setup

Assume Harpie deploys Vault contracts on both:

```text
Ethereum
Polygon
```

Both deployments use:

```text
Same Vault address
Same serverSigner
```

Alice wants to update her recipient address to:

```text
0xBob
```

#### Step 1: Signature Generation

Harpie's server signs:

```text
(
    Alice,
    Bob,
    expiry,
    VaultAddress
)
```

Notice:

```text
chainId is NOT included
```

Alice receives signature:

```text
Signature S
```

#### Step 2: Use Signature on Ethereum

Alice calls:

```solidity
changeRecipientAddress(
    S,
    Bob,
    expiry
);
```

The transaction succeeds.

The contract records:

```solidity
_isChangeRecipientMessageConsumed[data] = true;
```

Therefore the signature cannot be reused again on Ethereum.

#### Step 3: Replay on Polygon

Polygon maintains separate contract storage.

Therefore:

```solidity
_isChangeRecipientMessageConsumed[data]
```

is still:

```text
false
```

on Polygon.

Alice can submit the exact same signature:

```solidity
changeRecipientAddress(
    S,
    Bob,
    expiry
);
```

The signature verification succeeds because the signed hash is identical.

The recipient address changes again despite the authorization already being consumed on Ethereum.

The same authorization has now been replayed across multiple networks.

> **Analogy**: Imagine receiving a concert ticket that says which seat you can occupy and when the ticket expires, but it does not specify which venue it belongs to. If two venues use the same ticket format, the exact same ticket could be accepted at both locations even though it was intended for only one.

## Vulnerable Code Reference

### 1) Signature Hash Omits Chain ID

```solidity
bytes32 data = keccak256(
    abi.encodePacked(
        msg.sender,
        _newRecipientAddress,
        expiry,
        address(this)
    )
);
```

The signature is bound to:

```text
User
Recipient
Expiry
Contract Address
```

but not:

```text
Blockchain Network (chainId)
```

### 2) Replay Protection Exists Only Per Chain

```solidity
mapping(bytes32 => bool)
    private _isChangeRecipientMessageConsumed;
```

The mapping prevents reuse only within the storage of a single deployment.

Since every blockchain has independent storage:

```text
Ethereum consumed = true
Polygon consumed = false
```

the same signature remains reusable across networks.

### 3) Signature Validation Still Succeeds Cross-Chain

```solidity
require(
    data.toEthSignedMessageHash().recover(_signature)
        == serverSigner,
    "Invalid signature"
);
```

Because the signed data is identical across chains, the recovered signer remains valid everywhere the same signer is used.

## Recommended Mitigation

### 1. Include Chain ID in Signed Data (Primary Fix)

Bind signatures to a specific blockchain:

```solidity
bytes32 data = keccak256(
    abi.encodePacked(
        block.chainid,
        msg.sender,
        _newRecipientAddress,
        expiry,
        address(this)
    )
);
```

This ensures signatures generated on one network cannot be replayed on another.

### 2. Adopt Strong Domain Separation

Every signature should be uniquely scoped to:

* Chain
* Contract
* User
* Action
* Expiry

This minimizes the possibility of unintended authorization reuse.

### 3. Consider EIP-712 Structured Signatures

Using EIP-712 naturally incorporates domain separation through:

```text
Chain ID
Verifying Contract
Domain Name
Version
```

making cross-chain replay attacks significantly harder.

### 4. Add Cross-Chain Replay Tests

Introduce tests verifying that:

* A signature generated on Chain A succeeds on Chain A.
* The same signature fails on Chain B.
* Replay protection works as expected across deployments.

## Pattern Recognition Notes

* **Missing Domain Separation**: Signatures should always contain enough context to uniquely define where they are valid.
* **Cross-Chain Replay Risk**: Any signature lacking chain-specific information may become valid on multiple networks.
* **Contract Address Alone Is Not Sufficient**: A contract can exist at the same address on different chains.
* **Replay Protection Is Local State**: Storage-based replay protection does not extend across blockchains.
* **Prefer EIP-712**: Structured signatures provide safer domain separation than ad-hoc message hashing.
* **Think About Authorization Scope**: Whenever a signature authorizes an action, ask: "Where should this authorization be valid?" and explicitly encode that answer into the signed message.

## Quick Recall (TL;DR)

* **Bug**: `changeRecipientAddress()` signs data without including `block.chainid`.
* **Impact**: A valid signature from one network can be replayed on another network.
* **Why Replay Protection Fails**: `_isChangeRecipientMessageConsumed` only exists in the storage of a single chain.
* **Fix**: Include `block.chainid` in the signed message (or migrate to EIP-712 domain separation).
* **Core Lesson**: Signatures must be bound not only to users and contracts, but also to the blockchain where they are intended to be used.
